# Журнал рішень AiMay

## Meta-decisions

Правила, за якими цей журнал ведеться — не що вирішено, а як:

- Рішення фіксуються назавжди; ніщо не видаляється заднім числом.
- Відхилені пропозиції лишаються частиною історії проєкту, з
  поясненням, чому їх відхилено — не тільки прийняті.
- Архітектурні інваріанти (`FORMAL_SEMANTICS.md` §5) переважають
  зручність реалізації.
- Реальний сценарій передує розширенню мови; мова не розширюється
  тому, що розширення можна собі уявити корисним.

## Рішення №1–15 (Генеральний Реєстр, v0.1)
Базові команди, двофазний Bridge (задекларовано, реалізовано у v0.2),
патерни Dissonance (13), FORK як пошук найкращого шляху без клонування
Intent (14), динамічний HOLD з decay довіри (15).

## Рішення №16 — v0.1.1 (баг-фікси)
- CHECK.best_match більше не хардкод; читає last_fork_results від FORK.
- AWAIT коректно парсить сигнал-список [A, B, C] окремо від timeout.

## Рішення №17 — v0.2 (Protocol of Intent, keeper: Джем)
- ContextNode/ContextGraph замість dict, BFS-резонанс: score = trust * exp(-alpha*distance).
- AiMaySpaceSimulator — персистентний граф простору між командами.
- Bridge — двофазний: PENDING -> FULFILLED/BROKEN, engine попереджає про незакриті мости.
- BIND — ізоляція: поки Bridge відкритий, зв'язана пара захищена від повторного BIND.
- Command registry: плагінна архітектура через декоратор @command(name).

## Відкрите питання
Розділ 5 протоколу ("Абстракція виконання") обірваний на "кожна команда
приймає (engine, intent, args) і повертає с..." — сигнатура прийнята
як (engine, args) за аналогією з v0.1, повернене значення не
використовується диспетчером. Якщо Джем має повний текст із форматом
повернення (наприклад, CommandResult), потрібно звірити й, можливо,
допрацювати.

## Рішення №18 — v0.3 (command set expansion)
Голосування: Гепа (проти нових команд, за атрибути), Джем + Рей (за TASK
як команду з gating-поведінкою для батьківського Active). Вирішено на
користь Джема/Рея — поведінка "батько лишається Active, поки TASK не
resolved" не виражається через атрибут DECL, отже проходить літмус-тест
мови: нова команда додається лише тоді, коли її семантика не може бути
виражена існуючим набором команд.

- **TASK** — прийнято. 12-та команда. Дочірній Intent: власний Purpose,
  успадкований Origin, spawned_by = батько. Не може мутувати Purpose
  батька (вже гарантовано §5.1, окремий механізм не потрібен).
- **ADOPT** — прийнято (запропоновано Гепою). Змінює spawned_by, не
  чіпає origin. Семантика — stewardship/guardianship (кураторство,
  відповідальність за подальшу долю Intent), не ownership: історія й
  походження не переписуються, тільки те, хто тепер відповідає.
  Відрізняється від BIND: BIND — відношення зв'язку, ADOPT —
  відношення відповідальності.
- **QUERY** — прийнято (запропоновано Реєм). Read-only інтроспекція
  ContextGraph: strongest resonance, all bonds, memory above trust X.
  QUERY never mutates ContextGraph — жодних побічних ефектів.
  Проходить тест "неможливо виразити наявними командами".
- **Серіалізація** — НЕ окрема команда. `SEAL "file.obj"` (експорт),
  `DECL X from "file.tsk"` (відновлення) — розширення синтаксису
  наявних команд замість SERIALIZE/RESTORE, запропонованих Реєм.
- **MERGE** — відхилено до появи конкретного кейсу. Спекулятивне
  розширення без задачі, що його вимагає; сам Рей не навів приклад,
  де 11 команд не вистачає сьогодні. Правило від Гепи, зафіксоване
  цим кейсом: сценарій народжує команду, не навпаки — команда не
  додається лише тому, що її можна уявити корисною.

Потрібне оновлення: `docs/FORMAL_SEMANTICS.md` §2 (Lifecycle) має
зафіксувати правило "parent Active gated by unresolved TASK children"
— це нова автоматна поведінка, не описана в поточній редакції.
Заплановано на наступний раунд формалізації.

## Рішення №19 — v0.3 (ADOPT, QUERY)
- **ADOPT** — реалізовано. Invariant (Гепа): ADOPT never changes
  identity. It changes only responsibility. Перевірено регресійним
  тестом на два покоління (A → X через TASK, B adopts X, X → Y через
  TASK): origin і purpose незмінні транзитивно, spawned_by коректний
  на кожному рівні, ADOPT над X не відкочується наступним TASK від X.
- **QUERY** — реалізовано. Read-only: bonds | memory [threshold] |
  resonance. Перевірено найсуворішим тестом з наявних — побайтовий
  знімок усього стану рушія (Intent status/origin/purpose/spawned_by/
  bonds/children/context-graph, простір, bridges, кількість
  dissonances) до і після серії QUERY-викликів, включно з невідомим
  Intent і неправильними параметрами. Знімки ідентичні.

## Спостереження (не рішення): engine.current
Тест ADOPT на два покоління показав, що "хто зараз активний Intent"
— внутрішній стан інтерпретатора (`engine.current`), не поняття мови.
Немає команди "перемкнути фокус", TASK/BIND/etc завжди діють від
імені `engine.current`. Свідомо НЕ вводимо команду під це зараз:

> Current focus is an implementation concern, not yet a language
> concern.

Лежить тут до появи першого реального `.aim`-сценарію, який дійсно
неможливо виразити без явного перемикання фокусу — той самий принцип
"сценарій народжує команду", застосований до самого себе.

## Рішення №20 — Phase A closure (Path B, Execution Space, methodology)

**AiMay specifies meaning. Engines specify execution.** (Ардан, Path
B). This decision defines the architectural boundary of AiMay. Специфікація фіксує семантичні гарантії поведінки, не конкретні
алгоритми чи формули. Reference Engine — одна коректна реалізація,
не еталон для побітового відтворення. Якщо колись знадобиться
сертифікація сумісності незалежних рушіїв — це майбутній окремий
`CONFORMANCE.md` (Гепа), не зараз, за тим самим принципом, що й
`ARCHITECTURE.md`: документ чекає на реальну потребу, не пишеться
наперед.

**Execution Space** (Гепа, тест ізоляції: чи властивість Intent
зберігає сенс, якщо видалити всі інші Intent):
- `children`, `spawned_by` — підтверджено: належать Execution Space,
  не Intent. Мають сенс лише через посилання на інший Intent.
- `bonds` — **not yet**, не "ніколи". Перевірка всіх наявних `.aim`
  прикладів показала: жоден BIND не зв'язує два різні Intent, лише
  `self` з вузлом власного простору. Bonds лишаються частиною Intent
  доти, доки не з'явиться реальний сценарій міжінтентного BIND —
  формулювання навмисно слабше за "belongs to Intent", щоб не
  забороняти майбутнє випадково.

**Формула резонансу** — НЕ Hard Invariant. Обговорення показало:
навіть фіксація класу функції (експонента) не гарантує однаковий
`best_match` між рушіями з різними коефіцієнтами — детермінізм між
незалежно сконфігурованими рушіями недосяжний в принципі, тому
специфікація гарантує лише впорядкування (ближче/довірливіше —
вище), не конкретне обчислення. Внесено в `FORMAL_SEMANTICS.md` §3.

**Методологічні правила цього раунду:**
- *A question must first find its layer before it finds its answer*
  (Гепа). Шари: Vision → Design Philosophy → Formal Semantics →
  Reference Engine → (майбутній) Conformance.
- **Never generalize from an implementation capability to a language
  capability.** Те, що рушій технічно дозволяє (BIND між будь-якими
  токенами), не означає, що мова це використовує чи вимагає. Це
  правило одного рівня зі "сценарій народжує команду" — не окреме
  спостереження цього раунду.
- Один рушій має бути детермінованим: той самий `.aim`, той самий
  вхід, та сама конфігурація → той самий результат. Це вимога
  коректності одного рушія, не вимога однаковості між рушіями.

## Рішення №21 — Independent Semantic Audit (Path B verification)

Незалежний аудит `FORMAL_SEMANTICS.md` проведено на чотирьох різних
моделях (Stone/GPT, Kimi, Grok, Le Chat), кожна без контексту історії
проєкту — отримали лише специфікацію й журнал рішень, без пояснення
наміру команди. Протокол: рев'юер шукає семантичну неоднозначність,
кожна знахідка оскаржується доти, доки не зведеться до уявної моделі
виконання (і тоді відхиляється) або встоїть на самому тексті.

**Результат:** усі чотири моделі незалежно збіглися на тому самому
периметрі свободи, який уже зафіксовано в Decision №20 і Open
Questions — нової семантичної неоднозначності поза цими межами не
знайдено жодною. Це незалежне (не самим авторами) підтвердження
Path B.

**Одне не спростоване, але й не підтверджене питання:** життєвий
цикл Dissonance після зникнення однієї сторони — знайдено лише
Стоуном, три інші моделі на нього не вийшли. Внесено в Open Questions
§6 як кандидат, не як визнану прогалину.

Методологічна примітка: підсумкове memo про аудит спершу узагальнило
лише висновок, без сирого транскриту. Транскрипт запитано і наданий
окремо — саме він, не memo, є первинним доказом. На майбутнє: аудити
такого роду документуються разом із транскриптом, не тільки
підсумком.
